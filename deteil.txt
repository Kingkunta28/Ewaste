1. PROJECT SUMMARY
------------------

The Smart E-Waste Collection System is a role-based web application for
coordinating electronic-waste pickup requests. Residents submit pickup
requests, administrators assign collectors and monitor operations, and
collectors complete assigned pickups.

The system is a single repository containing:

- A Django JSON API and Django administration site.
- A React single-page application built with Vite.
- PostgreSQL support for production through Neon.
- SQLite fallback for local development and isolated tests.
- A professional monthly PDF performance report generated with ReportLab.
- Render deployment configuration for the backend.
- Vercel deployment configuration for the frontend.

The application currently supports three roles:

- User: registers, signs in, maintains a profile, creates pickup requests,
  and tracks request status.
- Admin: views users and collectors, registers collectors, assigns requests,
  changes request status, views dashboard statistics, and exports reports.
- Collector: views only requests assigned to their account, opens pickup
  addresses in an embedded map, and marks collections as completed.

4. DATABASE AND DOMAIN MODEL
----------------------------

4.1 Django User

The system uses django.contrib.auth.models.User. Normal registrations use the
email address as the username. Existing username-based accounts remain
supported during login.

Relevant standard fields include:

- username
- email
- first_name
- last_name
- password hash
- is_active
- is_staff
- is_superuser
- date_joined
- last_login

4.2 UserProfile

UserProfile has a one-to-one relationship with Django User.

Fields:

- role: user, admin, or collector.
- phone: optional text, maximum 20 characters.
- address: optional text, maximum 255 characters.

A post-save signal guarantees that every Django user has a profile. Staff and
superuser accounts are automatically assigned the admin profile role.

4.3 EWasteRequest

Fields:

- user: request owner.
- item_type: maximum 120 characters.
- quantity: positive integer, default 1.
- condition: optional, maximum 50 characters.
- brand: optional, maximum 80 characters.
- pickup_address: maximum 255 characters.
- pickup_date: requested collection date.
- notes: optional free text.
- status: pending, assigned, completed, or cancelled.
- assigned_collector: optional Django User foreign key.
- assigned_at: assignment timestamp.
- completed_at: completion timestamp.
- created_at and updated_at: audit timestamps.

Model validation enforces the following invariants:

- An assigned collector must have the collector role.
- Assigned and completed requests must have a collector.
- A completed request receives a completion timestamp.
- full_clean() runs before every EWasteRequest save.

4.4 Removed model

ServiceRating existed in migration 0002 and was removed in migration 0003.
It is not part of the current application model.

4.5 Features not represented by database models

The project does not currently have structured models for:

- Collection centers.
- Districts or regions.
- GPS coordinates.
- Login/audit events.
- Notifications.
- Equipment shortages or technical support incidents.
- Verified material weights or recycling output.

Reports must not treat these items as measured database facts.


5. REQUEST LIFECYCLE AND ROLE WORKFLOW
--------------------------------------

Normal lifecycle:

1. A user submits a request.
2. The request is created with pending status.
3. An admin selects a collector and assigns the request.
4. Assignment stores the collector, assigned status, and assigned_at time.
5. The assigned collector sees the request in their task list.
6. The collector or admin marks the request completed.
7. Completion stores completed status and completed_at time.

User permissions:

- Can list only requests owned by the logged-in user.
- Can create requests.
- Can view only owned requests.
- Can edit only owned pending requests.
- Cannot change workflow status.

Collector permissions:

- Can list only requests assigned to the logged-in collector.
- Can view only assigned requests.
- Can update only assigned requests.
- Cannot create pickup requests.
- Cannot assign collectors.

Admin permissions:

- Can list all pickup requests.
- Can assign collectors.
- Can update supported statuses.
- Can view registered users and collectors.
- Can register collector accounts.
- Can access global dashboard statistics.
- Can generate monthly PDF reports.


6. API REFERENCE
----------------

Base path: /api/

Authentication:

- POST auth/register/
  Creates a standard user account.

- GET auth/csrf/
  Initializes the CSRF cookie used by unsafe requests.

- POST auth/login/
  Authenticates with email or legacy username and creates a session.

- POST auth/forgot-password/
  Directly changes a password using email and a new password.
  This endpoint is not safe for production until verification is added.

- POST auth/logout/
  Ends the current Django session.

Current account:

- GET me/
  Returns the authenticated user and profile role.

- GET profile/
  Returns the authenticated profile.

- PATCH profile/
  Updates name, email, phone, and address.

Pickup requests:

- GET requests/
  Returns rows filtered by role.

- POST requests/
  Creates a request; standard users only.

- GET requests/<request_id>/
  Returns one request if permitted by role and ownership.

- PATCH requests/<request_id>/
  Updates an owned pending request.

- POST requests/<request_id>/assign/
  Assigns a collector; admin only.

- POST requests/<request_id>/status/
  Updates workflow status with role and assignment checks.

Administration:

- GET collectors/
  Lists collector accounts; admin only.

- GET users/
  Lists standard user accounts; admin only.

- POST collectors/register/
  Creates a collector account; admin only.

- GET dashboard/stats/
  Returns global request counts; admin only.

- GET reports/monthly-pdf/?month=YYYY-MM
  Returns an attachment PDF; admin only.

Errors are normally returned as JSON in this form:

  {"error": "Message"}


7. AUTHENTICATION, CSRF, CORS, AND COOKIES
-----------------------------------------

Authentication uses Django sessions rather than JWT tokens.

Frontend behavior:

- All fetch requests use credentials: include.
- Unsafe methods first request /api/auth/csrf/ if no csrftoken exists.
- Unsafe requests send X-CSRFToken.
- A cached user summary is stored in localStorage for initial rendering.
- /api/me/ remains the authoritative session check at startup.

CORS behavior:

- SimpleCorsMiddleware supports exact origins and strict wildcard subdomains.
- Credentials are allowed.
- Allowed methods include GET, POST, PATCH, PUT, DELETE, and OPTIONS.
- Allowed headers include Content-Type, Authorization, and X-CSRFToken.

Production cookie behavior:

- Session and CSRF cookies are Secure.
- SameSite defaults to None in production.
- The deployed Vercel frontend is a trusted CSRF origin.
- HSTS and HTTPS redirect settings are enabled in production.

10. VALIDATION AND INPUT RULES
------------------------------

Registration:

- First name, last name, email, and password are required.
- Email uniqueness is checked case-insensitively.
- Django password validators are applied.

Pickup creation:

- Item type, address, and pickup date are required.
- Quantity must be an integer of at least 1.
- Date must use YYYY-MM-DD and cannot be in the past.

Pickup editing:

- Only the owner can edit.
- Only pending requests can be edited.
- Empty item type and address are rejected.
- Quantity and date are revalidated.
- Notes, phone, address, and other optional values can be cleared.

Assignment and completion:

- Collector ID must be numeric and must reference a collector role.
- Completed and cancelled requests cannot be assigned.
- A request cannot be completed without an assigned collector.

14. DEPLOYMENT WORKFLOW
-----------------------

14.1 Render backend

render.yaml defines one Python web service:

- Build command: bash ./build.sh
- Start command: gunicorn backend.wsgi:application
- Production mode enabled.
- Vercel origin configured for CORS and CSRF.
- Secure SameSite=None cookies configured.
- DATABASE_URL and MIGRATION_DATABASE_URL expected as Render secrets.

build.sh performs:

1. pip install -r requirements.txt
2. collectstatic
3. migrations, using MIGRATION_DATABASE_URL when available
4. idempotent superuser creation/update when credentials are present

14.2 Vercel frontend

frontend/vercel.json defines:

- Framework: Vite.
- Install command: npm install --include=dev.
- Build command: npm run build.
- Output directory: dist.
- /api/* rewrite to the Render backend.
- SPA fallback to index.html.

Vercel should use frontend/ as the Root Directory.

14.3 Database consistency

Local .env is ignored by Git and does not update Render. To make production use
a new Neon database, update DATABASE_URL and MIGRATION_DATABASE_URL in the
Render dashboard and redeploy. Vercel does not connect directly to PostgreSQL;
it communicates through the Render API.


15. SECURITY AUDIT AND PRIORITIZED RISKS
----------------------------------------

CRITICAL - Rotate and remove committed administrator password

render.yaml currently contains DJANGO_SUPERUSER_PASSWORD as a literal value.
This is a committed credential and should be considered compromised.

Required action:

1. Rotate the password immediately in Render.
2. Change render.yaml back to:

     - key: DJANGO_SUPERUSER_PASSWORD
       sync: false

3. Remove the credential from Git history if the repository is public.
4. Prefer a unique high-entropy password and enable operational access control.

CRITICAL - Replace direct password reset

/api/auth/forgot-password/ resets a password using only an email address and a
new password. Anyone who knows a registered email can take over that account.

Required action:

- Use Django password-reset tokens delivered to verified email, or a secure OTP
  with expiry, one-time use, throttling, and non-enumerating responses.

HIGH - Rotate exposed test credentials

TEST USERS.txt is tracked and contains passwords. These accounts must be
development-only. Never reuse their passwords in production. Remove or sanitize
the file before public distribution.

HIGH - Do not expose exception details from PDF generation

The PDF endpoint returns str(exc) to authenticated admins. Log exceptions on the
server and return a generic error message to avoid disclosing internal details.

MEDIUM - Improve throttling storage

Authentication throttling uses Django's default cache. Without a shared cache,
limits are per process and disappear after restarts. Use Redis or another shared
production cache and consider django-ratelimit or equivalent controls.

MEDIUM - Harden account identity rules

Email uniqueness is enforced in view logic but not by a case-insensitive database
constraint. Concurrent requests can still create duplicates. Consider a custom
user model with normalized unique email.

MEDIUM - Remove legacy unreachable report code

Delete the old report implementation after the early return in views.py and
remove its unused imports.

MEDIUM - Add frontend and security tests

Add tests for routing, form submission, API failures, CSRF, CORS, password reset,
PDF generation, admin authorization, and collector transitions.

LOW - Stop tracking generated frontend dist assets

Six files under frontend/dist are tracked. Vercel builds from source, so these
generated assets are unnecessary in Git and create noisy hashed-file changes.
Add frontend/dist/ to .gitignore and remove it from tracking if no other deploy
process depends on committed build output.


16. FUNCTIONAL LIMITATIONS AND TECHNICAL DEBT
---------------------------------------------

- Collection centers are not implemented as a model or workflow.
- Notifications are not implemented; the remaining admin/user bell is visual.
- Search input and Ctrl+K label have no search behavior.
- Admin monthly bar chart uses fixed sample values.
- Landing statistics and member counts are static marketing content.
- Remember me does not control Django session duration.
- Explore recycling tips links to #tips, but no verified tips section workflow
  is connected from the dashboard.
- Export Data calls window.print(); it does not export CSV or spreadsheet data.
- Collector Completed Tasks navigation scrolls to the map card.
- Environmental impact uses estimates, not measured device weights.
- Address strings are used for maps and geography; there are no coordinates.
- There is no cancellation-reason, issue-ticket, or equipment-tracking module.
- There is no real-time refresh, WebSocket layer, or notification service.
- App.jsx is densely formatted and mixes routing, session, and layout concerns.
- views.py contains many function-based endpoints in one large module.
